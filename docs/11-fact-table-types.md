# Module 11: Fact Table Types

## Goal

The goal of this module is to understand that not all fact tables are the same.

In dimensional modelling, a fact table stores business measurements or events. The type of fact table depends on the business process and the grain.

The four main fact table types are:

```text
1. Transaction fact
2. Periodic snapshot fact
3. Accumulating snapshot fact
4. Factless fact
````

## 1. Transaction fact table

A transaction fact table stores individual business events.

The grain is usually:

```text
One row per event.
```

Examples:

```text
One row per bank transaction
One row per card payment
One row per loan repayment
One row per product sale
One row per insurance claim
```

## Banking example: `fact_transaction`

```text
fact_transaction
- transaction_key
- transaction_reference
- date_key
- account_key
- product_key
- branch_key
- channel_key
- transaction_type_key
- transaction_amount
- transaction_count
```

Grain:

```text
One row per posted account transaction.
```

This type is best when analysing things that happened.

Questions it can answer:

```text
Total transaction value by month
Transaction count by channel
Card payments by customer segment
Fees generated by product
Transaction volume by branch
```

## 2. Periodic snapshot fact table

A periodic snapshot fact table stores measurements at regular time intervals.

The grain is usually:

```text
One row per thing per time period.
```

Examples:

```text
One row per account per day
One row per customer per month
One row per loan account per month
One row per product per branch per day
```

## Banking example: `fact_daily_account_balance`

```text
fact_daily_account_balance
- date_key
- account_key
- product_key
- branch_key
- account_status_key
- opening_balance
- closing_balance
- available_balance
- account_count
```

Grain:

```text
One row per account per day.
```

This is different from a transaction fact.

A transaction is an event.

A balance is a state at a point in time.

So they should not be mixed in one fact table.

Questions it can answer:

```text
What was the closing balance yesterday?
What was the average balance by product this month?
What was the total balance by branch at month-end?
How did balances trend over time?
```

Important:

```text
Balances are usually semi-additive.
```

You can sum balances across accounts for one day, but you should not blindly sum balances across many days.

## 3. Accumulating snapshot fact table

An accumulating snapshot fact table tracks a process that has stages.

The grain is usually:

```text
One row per business process instance.
```

The row gets updated as the process moves through stages.

Examples:

```text
One row per loan application
One row per insurance claim
One row per customer onboarding journey
One row per order fulfilment process
```

## Banking example: loan application pipeline

A loan application may move through these stages:

```text
Application submitted
Documents received
Credit check completed
Approved or declined
Loan account opened
Funds disbursed
```

Model:

```text
fact_loan_application_pipeline
- application_key
- customer_key
- product_key
- submitted_date_key
- documents_received_date_key
- credit_check_date_key
- decision_date_key
- disbursement_date_key
- application_amount
- approved_amount
- days_to_decision
- days_to_disbursement
- application_status
```

Grain:

```text
One row per loan application.
```

Questions it can answer:

```text
How long does loan approval take?
How many applications are stuck before credit check?
What percentage of applications are approved?
What is the average time from submission to disbursement?
```

## 4. Factless fact table

A factless fact table records that something happened or that a relationship existed, but it has no numeric measure.

The grain is still important.

Examples:

```text
One row per student attending a class
One row per customer eligible for an offer
One row per account enrolled in a campaign
One row per customer login event
One row per suspicious rule trigger
```

## Banking example: campaign eligibility

```text
fact_campaign_eligibility
- date_key
- customer_key
- campaign_key
- product_key
```

Grain:

```text
One row per customer per campaign per eligibility date.
```

There may be no amount.

The fact is simply:

```text
This customer was eligible for this campaign on this date.
```

## Banking example: fraud rule trigger

```text
fact_fraud_rule_trigger
- transaction_key
- fraud_rule_key
- date_key
```

Grain:

```text
One row per fraud rule triggered per transaction.
```

A transaction can trigger multiple fraud rules, so this table records those occurrences.

## Comparison

| Fact type                  | Grain example                          | Best for                        |
| -------------------------- | -------------------------------------- | ------------------------------- |
| Transaction fact           | One row per transaction                | Events that happen              |
| Periodic snapshot fact     | One row per account per day            | Regular state measurements      |
| Accumulating snapshot fact | One row per application                | Multi-stage processes           |
| Factless fact              | One row per occurrence or relationship | Events without numeric measures |

## Banking examples together

A proper banking warehouse may contain:

```text
fact_transaction
- One row per posted account transaction

fact_daily_account_balance
- One row per account per day

fact_loan_application_pipeline
- One row per loan application

fact_campaign_eligibility
- One row per customer per campaign per date

fact_fraud_rule_trigger
- One row per fraud rule triggered per transaction
```

Each table has a different grain and purpose.

## Common mistakes

### Mistake 1: Mixing transaction facts and snapshot facts

Bad:

```text
fact_account_activity
- transaction_amount
- closing_balance
```

Problem:

```text
transaction_amount = event-level
closing_balance = account-day-level
```

Different grains need different fact tables.

Better:

```text
fact_transaction
fact_daily_account_balance
```

### Mistake 2: Summing balances over time

Bad:

```sql
SELECT
    month_key,
    SUM(closing_balance) AS monthly_balance
FROM fact_daily_account_balance
GROUP BY month_key;
```

This usually gives a meaningless number.

Better options:

```text
Use end-of-month balance
Use average daily balance
Use latest balance in selected period
```

### Mistake 3: Forcing every fact table to have an amount

Some useful fact tables do not have numeric measures.

Examples:

```text
Customer attended event
Fraud rule triggered
Customer eligible for campaign
Student enrolled in course
```

These can still be valid fact tables.

### Mistake 4: Vague fact table names

Weak:

```text
fact_status
fact_activity
fact_summary
```

Better:

```text
fact_loan_application_pipeline
fact_daily_account_balance
fact_fraud_rule_trigger
```

A good fact table name should reveal the business process and grain.

## How to choose the fact type

Use this decision pattern:

```text
Event happened?
→ Transaction fact

State measured regularly?
→ Periodic snapshot fact

Process with stages?
→ Accumulating snapshot fact

Occurrence with no numeric measure?
→ Factless fact
```

## Key takeaways

* Not all fact tables are the same.
* Transaction facts record events.
* Periodic snapshots record states at regular intervals.
* Accumulating snapshots track process lifecycles.
* Factless facts record occurrences or relationships without numeric measures.
* Grain must be defined before choosing the fact table design.
* Transactions and balances should usually be separate fact tables.
